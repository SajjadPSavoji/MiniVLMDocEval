This is roughly 39 percentage points lower than its performance on DocVQA (89.3), establishing visual table QA as the primary frontier bottleneck at this scale.The relative success of Qwen3.5-0.8B over its competitors is explained by its specialized image-processing and sequence-mixing design. Standard tiling pipelines, such as the one used by InternVL3-1B, split high-resolution images along artificial grid boundaries. While tiling is highly effective for localized text extraction, it splits table rows and columns across separate processing tiles, breaking the spatial alignment of the grid. In contrast, Qwen3.5-0.8B's native dynamic resolution allows arbitrary aspect ratios and resolutions, avoiding the grid division errors of tiling.This visual capability is reinforced by the hybrid backbone of the target model, which interleaves 18 recurrent Gated DeltaNet layers with 6 standard Attention layers in a 3:1 sequence mixing layout. This design maintains subquadratic memory growth while preserving the long-context representations needed to track tabular indices.Despite these structural advantages, the target model still fails to resolve 49.9% of visual table questions. Analyzing the target model's failure modes across the four sub-domains of the benchmark—Visual WikiTableQuestions (VWTQ), Synthesized Visual WTQ (VWTQ-Syn), Visual TabFact (VTabFact), and FinTabNetQA—reveals four distinct failure categories :VWTQ (Visual WikiTableQuestions) and Row-Column Indexing: These tables feature diverse layouts and cell types. The primary failure mode is neighborhood drift during visual processing, where the model's visual attention shifts to adjacent cells when trying to locate cell content by row-column coordinates. This drift is caused by the loss of 2D spatial information when visual tokens are serialized into a 1D sequence before entering the language backbone.VWTQ-Syn (Synthesized Visual WTQ) and Color-Typography Overload: These tables use randomized styles, including alternating row colors and custom borders. While the model is generally robust to basic layout variations, it struggles when text-mixed cells are combined with dense background shading, resulting in tokenization errors and layout confusion.VTabFact (Visual TabFact) and Logic/Verification Failures: This subset requires verifying tabular statements as True or False. The model fails at multi-hop comparisons and relational logic across multiple columns, indicating a failure to align visual representations with the tabular structure.FinTabNetQA (S&P 500 Financial Sheets) and Scale Disconnects: These financial reports feature complex layouts and multi-column hierarchies. The model frequently fails to associate numerical cell values with scale units (such as thousands or millions) located in distant column headers or table legends, leading to incorrect calculations.These failure modes can be divided into factors that are fixable through post-training and those that are baked into the model's pretraining and architecture.Fixable through Post-Training: Row-column tracking, scale association, multi-step numerical processing, and relational logic can be addressed during post-training. Providing structured rationales and targeted fine-tuning data helps the model learn the spatial coordinates of rows and columns.Baked into Pretraining and Architecture: The raw visual representation limit of the vision encoder (e.g., SigLIP or ViT patch size), the capacity of the projection layer, and the core mathematical processing limits of a 0.8B parameter language model are hard constraints. In Gated DeltaNet, the state matrix $S_t \in \mathbb{R}^{H \times K \times V}$ is updated at each step using a gated delta rule :$$S_t = \alpha_t S_{t-1} (I - \beta_t k_t k_t^\top) + \beta_t v_t k_t^\top$$where $\alpha_t$ and $\beta_t$ are learnable decay and update gates, and $k_t$ and $v_t$ are key and value projections.Standard fine-tuning methods like LoRA modify the model's weight projection matrices. In contrast, $S_0$ Tuning freezes all weight parameters and optimizes only the initial state tensor $S_0^{(\ell)}$ for each recurrent layer $\ell$. By training $S_0$ on correct tabular solution trajectories, the model is steered toward structured processing paths from the very first token, with zero inference overhead.The optimization objective uses a completion-only cross-entropy loss with $L_2$ regularization :$$\mathcal{L}(S_0) = \frac{1}{N} \sum_{i=1}^N \text{CE}(y_i^{\text{comp}}, f_\theta(x_i; \alpha S_0)) + \lambda \sum_{\ell=1}^L \|S_{0}^{(\ell)}\|_2^2$$where prompt tokens are masked, and $\alpha$ is a constant scaling factor (set to 0.07 for Gated DeltaNet). While $S_0$ Tuning is highly efficient, it acts primarily as a steering mechanism and cannot inject new visual features. Therefore, it is best used alongside a lightweight, structured LoRA.Methodology 4: Error-Targeted Data AugmentationTo address the model's specific failure modes on complex tables, this methodology generates synthetic tables matching those exact structural challenges. For example, the pipeline can render financial tables with mismatched units or nested headers using specialized tools like Misata or LaTeX renderers. While this targeted data generation directly addresses the model's weaknesses, it requires a complex rendering pipeline and carries a risk of layout distribution shift.Catastrophic Forgetting Mitigation StrategyWhen specializing small VLMs on visual table tasks, there is a high risk of catastrophic forgetting. This forgetting can affect both downstream task capabilities (such as chart analysis and infographic interpretation) and pre-trained visual grounding abilities.Recent studies on multimodal fine-tuning show that catastrophic forgetting is primarily caused by over-optimization and task-specific overfitting. When a model is fine-tuned on specialized task data, standard full-parameter updates with high learning rates (e.g., $1 \times 10^{-5}$) severely degrade performance on out-of-distribution visual inputs. However, restricting the updates through parameter-efficient methods (LoRA) or using a low learning rate (e.g., $1 \times 10^{-6}$ for full backbone updates, or $1 \times 10^{-4}$ for LoRA) helps preserve the model's pre-trained capabilities.                 ----> Overwrites Pre-trained Visual Alignments (Severe Forgetting)

Kernel Compilation and Execution Errors: Qwen3.5 uses custom Triton kernels to compile Gated DeltaNet layers. Compiling these kernels can take longer on older GPUs (such as T4) and requires specific PyTorch and CUDA configurations. In low-resource environments (like Google Colab), developers must configure the training backend to bypass Megatron dependencies by setting the environment variable USE_MCORE_GDN=0. This falls back to the Transformers-compatible Gated DeltaNet implementation, ensuring stable training on a single GPU.Quantization Precision Collapse: Although 4-bit QLoRA is commonly used to save memory, QLoRA is not recommended for Qwen3.5 models. Gated DeltaNet's recurrent decay gates ($\alpha$) are highly sensitive to quantization errors under BitsAndBytes, which can cause training instability. Because Qwen3.5-0.8B is highly efficient, a standard bf16 (or fp16 on T4 GPUs) LoRA setup uses only ~3GB of VRAM during training. Verification and Sub-Capability Evaluation ProtocolsTo verify the fine-tuned model's capabilities and ensure it has not suffered from catastrophic forgetting, a two-part evaluation protocol is established.Sub-Skill EvaluationFirst, the model's table understanding is measured across four core sub-skills:Structure Parsing: Evaluated on the chemical table benchmark (ChemTable) using the TEDS-Struct (Tree-Edit-Distance for Table-Structure) metric. This measures how accurately the model recovers row and column relationships, independent of cell content.Cell Lookup and Value Retrieval: Measured using the 2Columns1Row benchmark. This benchmark tests the model's ability to locate a target cell in one column based on a key value in another column, directly measuring spatial coordination.Mathematical Aggregation: Evaluated on numerical reasoning subsets from FinTabNetQA and VWTQ. Accuracy is calculated using a relaxed accuracy metric ($\pm 5\%$ margin for numerical rounding errors).Fact Verification: Measured using the VTabFact subset, which contains True/False verification queries. Performance is evaluated using strict Exact Match (EM) accuracy.Regression Detection PipelineSecond, to detect regressions on general document tasks, the model is evaluated on the original evaluation pipeline using VLMEvalKit.
